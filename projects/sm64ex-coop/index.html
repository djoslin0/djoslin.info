sm64ex-coop

sm64ex-coop was a fork of the SM64 PC port that I created to add real-time multiplayer to Super Mario 64.

The original goal was to synchronize everything over the internet so that multiple people could play through the campaign online. The goal was eventually expanded to include adding comprehensive modding support to allow people to alter any aspect of the game. These days you can join a lobby with custom player models, levels, gamemodes, and almost anything you can imagine. While I started it, I was glad to see it grow into a community project. The project lives on as sm64coopdx, which was originally a fork, but carries the torch today.

Development history of my contributions

sm64ex-coop was born at the end of July 2020. The COVID lockdown left me with a lot of free time, and I had been following the SM64 decompilation project for a while. The decompilation team had released a version of SM64 that ran natively on PC as C code, which caught my interest.

Part of my motivation came from being disappointed with Net64. It only synchronized basic player state like position, facing, and animation, so many world interactions stayed local to each player. I wanted coins, switches, objects, and scripted events to be shared between players, not just player positions.

I started without any planning. I crudely hacked in a second player and got the game to stop crashing. Before networking existed, I spent a few days altering the engine so that two players could exist at once: animations were separated, held objects began to track ownership, and most enemies and interactive objects were adjusted to be multiple-player-aware.

Once two players could exist locally, I started adding networking for players. Next was the long process of synchronized objects and events one by one: coins, teleports, star collection, chain chomp, moving platforms, puzzle switches, small enemies, and many level-specific behaviors. Many objects required different approaches, and different trade offs were made. Some required full state updates every frame, others only needed a single event when collected or destroyed. I went through each level one at a time and systematically synchronized each object in the level.

After the foundation was laid, I began creating an additional networking backend to use the Discord Game SDK to handle connections and NAT traversal. The Discord SDK solved many problems for users since most people are inexperienced with, or unable to, open ports on their router. But this ultimately proved to be a temporary solution as Discord would eventually shut down the service.

Then I added a "bubble system" to deaths in co-op. In the original game, dying sent a player back to the castle. But with the bubble system a dead player floats in a bubble until a teammate pops it to respawn them at that spot. I saw this in another SM64 romhack and thought it would be a fun feature to have, but would likely not have created it if I knew that lua modding would eventually be possible.

Next up was player-to-player interactions: attacks, knockback, bouncing off each other, and handling when two players try to grab the same object. The original engine assumed a single player, so these interactions required careful handling of ownership, collision, and animation.

Late 2020 into early 2021 focused on scaling and stability. I increased the player cap, improved late-join synchronization so new players arrived in a correct state, refined camera behavior, and hunted down crashes and desyncs that appeared as more people tried it. The bug squashing is never ending since the original code was already fragile. When so many assumptions are uprooted, the problems are sure to follow.

In mid-2021 I wrote DJUI, a custom UI framework. The original UI was very difficult to work with and extend. So I needed new a system for menus that could have input fields, checkboxes, server settings, player lists, chat, and console output. DJUI grew into a substantial part of the project... and while it has its own annoyances, it's a lot easier to work with than the alternative.

In early 2022 I started creating the Lua scripting API. I built a C-to-Lua bridge with sandboxing and hooks, and wrote tooling to generate bindings and documentation from the C code. That automation kept the Lua API accurate as the underlying codebase changed. I also added networked sync tables so Lua mods could declare shared state that the game would keep synchronized automatically. This change in the direction was foundational for what was to come since this mod would have mods be made for it as well. Initially I was concerned at the possibility that I would pour work into a very complicated system that wouldn't get used but I am very happy to find out that the concern was unfounded.

Around the same time I did a significant expansion of DynOS. DynOS was a system written by PeachyPeach to convert actor geolayout C source into binary files. It's essentially a very specific, stripped down, C compiler in a sense. Initially it allowed players to swap character models. Over the following weeks I extended it to compile collision data, full level scripts, moving textures, rooms, object trajectories, lights, custom skyboxes, and custom object behaviors. By the end, every vanilla level could be compiled through DynOS, and a mod author could ship a complete custom world as a mod.

Also in 2022 I worked on uncapped framerate and proper interpolation so the game could run at higher frame rates without physics or animation glitches. To accomplish this feature the renderer now interpolates object positions, animation frames, camera movement, and UI elements. After my work was implemented Altiami figured out the proper way to interpolate rotation matrices, and now you can run uncapped framerates seamlessly.

When Discord announced the end of its Game SDK, I decided to build my own replacement called CoopNet. CoopNet is a P2P networking library built on top of libjuice for peer discovery, NAT traversal, lobby management, and data transport. CoopNet allows the project to have full control over the connection layer.

Shortly afterward, I added lag compensation to make PvP feel responsive, and I built an Arena game mode inspired by Quake and Smash Bros that features custom levels, weapon pickups, and many sub-gamemodes like king of the hill and capture the flag.

From mid-2023 onward I focused heavily on stability work: crash fixes, bounds checking on network packets, safer Lua interaction, texture loading edge cases, and other subtle bugs. While this category of changes are unglamorous, they are still very important. However, around this point in development my interest in the project began waning.

The Coop Deluxe team forked the project as sm64coopdx in late 2023 during a long period in which I was taking a break. After sm64coopdx released a few versions the teams were merged and I stepped down as a lead developer from a lack of interest.

Around the time of the merger one of the last major things I worked on was moving all asset loading from the compile stage to the runtime stage. The original SM64 PC port required a ROM at build time so that assets could be extracted and baked into the executable. Every texture, sound sample, animation, collision mesh, vertex list, and skybox was sitting in the repository as pre-extracted C data. I designed and wrote the system to load all of that directly from the ROM at runtime instead. I believe Isaac further contributed to this feature as I passed it off to him.

Shoutouts

A project this size doesn't happen alone, and a few people put in work that I want to specifically acknowledge.

PeachyPeach wrote the original DynOS system that I later expanded. Without that foundation the whole model pack and custom level pipeline would not exist. She also kept contributing fixes and improvements throughout, makes cool mods, and generally knows what she's doing.

Isaac put in several hundred commits covering DynOS, DJUI, Lua, scroll textures, collision fixes, and frame pacing. A lot of his work is the kind of quiet, unglamorous bug fixing that a project this large desperately needs.

Agent X is likely the most active contributor to sm64coopdx after I stepped down, with close to 1,000 commits. His biggest single piece of work is the lighting engine, which he built, documented, and optimized over many iterations. He also took on a lot of Lua API exposure work, DJUI improvements, and general maintenance that kept the project moving.

fgsfds contributed the original 60 fps patch that the later interpolation layer was built upon. He also was the maintainer of sm64ex when coop was originally forked. He added changeable model colors, implemented the GL legacy backend for older hardware, and added SDL1 support.

EmeraldLockdown maintains the Mac port and has been doing a flurry of long-requested contributions as of late.

Prince Frizzy tackled performance and portability, most notably threaded audio support. He also contributed IPv6 support, dynamic model allocation, various security and sanity checks, updated the codebase to later refreshes, and a lot of build system work.

Fluffa made the Luigi and Wario player models. He also remade Toad, replacing the original model I had built. All of their models are well-made and fit the game's style without looking out of place.

Super Keeberghrh made the Waluigi model, which is also well-made and fits the game's style.

There are many others that made a sizable impact... too many to list. While this post is mainly focusing on my contributions, coop is a community project and felt compelled to include the contributions of others.

Conclusion

All told, of the roughly 4,500 commits since coop began, I pushed around 2,250 of them. The project was never meant to be a long-term thing, but I got a little bit carried away. While I'll probably contribute once in a while, I'm firmly in the backseat at this point.

I'm proud of what I've created and I praise the contributions of every person who added their work to the project. And I'm excited to see where the community continues to take it.
